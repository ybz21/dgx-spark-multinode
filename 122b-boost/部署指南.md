# 部署指南 · Qwen3.5-122B-A10B INT4 · 单节点 Boost

## 1. 前置条件

- DGX Spark (NVIDIA GB10，统一内存 ≥ 128GB)
- Docker + NVIDIA Container Toolkit
- 模型文件 `Qwen3.5-122B-A10B-int4-AutoRound-Intel`（14 个 safetensors 分片，合计 **72GB**）
- 镜像 `vllm-qwen35-v2`（19.1GB，基于 vLLM v0.19.1.dev + CUDA 13.2，ARM64）

**验证**：

```bash
nvidia-smi                                                          # GPU 可见
docker images vllm-qwen35-v2                                        # 镜像存在
du -sh ~/models/Qwen3.5-122B-A10B-int4-AutoRound-Intel               # 72G
ls ~/models/Qwen3.5-122B-A10B-int4-AutoRound-Intel/*.safetensors | wc -l  # 应为 15（14 shards + extra）
```

## 2. 跨节点同步（可选）

若需把已部署节点的镜像/模型复制到新节点，优先走高速网（ConnectX-7, 10.0.0.0/24）：

```bash
# 同步模型（走 rsync，72GB，增量续传）
rsync -aP --bwlimit=0 \
  -e 'ssh -c aes128-gcm@openssh.com' \
  ~/models/Qwen3.5-122B-A10B-int4-AutoRound-Intel/ \
  ai@10.0.0.X:~/models/Qwen3.5-122B-A10B-int4-AutoRound-Intel/

# 同步镜像（走 docker save | docker load）
docker save vllm-qwen35-v2 | \
  ssh -c aes128-gcm@openssh.com ai@10.0.0.X 'docker load'
```

高速网 <1ms 延迟，实测整机镜像流式 load 约 3–5 分钟。

## 3. 启动服务

配置文件：[`docker-compose.yaml`](./docker-compose.yaml)

```bash
mkdir -p ~/lm_scripts
cp docker-compose.yaml ~/lm_scripts/docker-compose-qwen-122b-a10b-ar-int4.yaml
cd ~/lm_scripts
docker compose -f docker-compose-qwen-122b-a10b-ar-int4.yaml up -d
```

**端口冲突**：容器使用 `network_mode: host` 直接监听宿主机 `30000`。部署前请停掉任何占用该端口的容器（例如旧的 Ray vLLM worker）：

```bash
docker ps | grep -E '30000|vllm'
docker stop <container> && docker rm <container>
```

## 4. 等待健康

首次启动要加载 72GB 权重 + 编译 FLASHINFER kernels，耗时 **10–15 分钟**。

```bash
# 观察状态（starting → healthy）
watch -n 5 'docker ps --filter name=vllm-qwen35 --format "{{.Status}}"'

# 看日志关键阶段
docker logs -f vllm-qwen35 2>&1 | grep -E 'Loading safetensors|Available KV cache|Application startup|ERROR'

# 就绪后
curl -sf http://localhost:30000/health && echo OK
curl -s http://localhost:30000/v1/models | jq '.data[0] | {id, max_model_len}'
# => {"id":"qwen3.5-122b-int4","max_model_len":131072}
```

## 5. 启动参数拆解

`docker-compose.yaml` 里 command 段逐项含义：

| 参数 | 值 | 作用 |
|---|---|---|
| `serve <path>` | `/models/Qwen3.5-122B-A10B-int4-AutoRound-Intel` | 模型路径 |
| `--served-model-name` | `qwen3.5-122b-int4` | 对外 API 模型名 |
| `--port` | `30000` | 监听端口 |
| `--max-model-len` | `131072` | 上下文上限（128K） |
| `--gpu-memory-utilization` | `0.90` | 显存预算比例 |
| `--reasoning-parser` | `qwen3` | 将 `<think>…</think>` 解析到 `reasoning` 字段 |
| `--attention-backend` | `FLASHINFER` | 长上下文注意力后端 |
| `--enable-auto-tool-choice` | — | 启用自动 tool choice |
| `--tool-call-parser` | `qwen3_coder` | Qwen3 工具调用解析 |
| `--trust-remote-code` | — | 加载模型自带代码 |
| `--speculative-config` | `{"method":"mtp","num_speculative_tokens":2}` | MTP 投机解码，k=2 |

### 5.1 显存预算

GB10 统一内存 128GB，`0.90` 约 115GB 预算：

| 条目 | 大小 |
|---|---|
| 权重 (INT4) | 67.35 GiB |
| KV Cache | 33.39 GiB (335,808 tokens，够 128k × 2.6 并发) |
| CUDA Graph | 0.70 GiB |
| **总** | ~101 GiB |

统一内存意味着 GPU 占用会挤占系统 RAM。如需给宿主其他容器让出内存，可降到 `0.80`（KV cache 约 20GiB，单会话 128k 够跑）。

### 5.2 MTP 投机解码

模型 `config.json` 中 `mtp_num_hidden_layers: 1`（只有 1 层 MTP）。`num_speculative_tokens` 设置：

| k | 接受率 | 净加速 | 说明 |
|---|---|---|---|
| 1 | ~85% | 1.5–1.7× | 保守 |
| **2** | **~60%** | **1.7–1.9×** | **当前默认** |
| 3+ | <40% | 可能反慢 | 同一 MTP 层被 reuse k 次，误差累积 |

### 5.3 Reasoning 开关

Qwen3 默认开启思考（`<think>`）。压测/短回答场景通过 API 字段关闭：

```json
{
  "model": "qwen3.5-122b-int4",
  "messages": [...],
  "chat_template_kwargs": {"enable_thinking": false}
}
```

思考内容以 `delta.reasoning`（或 `reasoning_content`）字段流式返回，非 `delta.content`。

## 6.1 长久稳定性测试

脚本：[`soak_test.py`](./soak_test.py)。混合负载 + 正确性探针 + 分时漂移监控，用于**长时间运行验证**（默认 4h，可跑整夜/整日）。

```bash
# 4 小时默认运行
python3 soak_test.py --base http://<host>:30000 \
    --duration 4h --concurrency 4 --out-dir soak_$(date +%Y%m%d_%H%M)

# 过夜 12 小时，后台跑
nohup python3 -u soak_test.py --duration 12h --out-dir soak_overnight \
    > soak.log 2>&1 &

# 目录产出
#   status.md     # 实时刷新的 Markdown 快照（60s 一次）
#   status.json   # 对应 JSON
#   events.log    # 错误 + probe 失败追加日志
#   summary.md    # 退出时生成的最终报告
```

**工作负载组成**（权重）：
- 70% short（512 tok prompt, 128 tok 输出）
- 20% medium（4k tok prompt, 256 tok 输出）
- 8% long（32k tok prompt, 200 tok 输出）— 用 `--disable-long` 可跳过
- 2% **probe**（已知答案的小测试题，如 `47+83=?`、'banana' 倒写）检测模型输出是否失常

**监控指标**：
- 滚动 p50/p95 TTFT、decode tok/s
- 每小时分桶 N / fails / p50/p95 TTFT / decode（检查**性能随时间漂移**、显存泄漏）
- 错误签名聚合（折叠地址/数字，归类同种错误）
- probe 通过率（检查模型是否产生垃圾输出）

**Ctrl-C / SIGTERM 会优雅退出并写 `summary.md`**。随时可以中断看结果。

## 7. 故障排查

### 启动卡在 "Loading safetensors"
属正常，14 分片每片 34–36 秒，共 ~9 分钟。期间 `health: starting`，`/health` 返回非 200。耐心等。

### 日志连续出现 `Failed to connect to GCS at address ...:6379`
说明是 **Ray worker 节点**在找 head。本方案为单节点，容器里不该跑 ray。检查是否误启了多节点 compose。确认用的是本目录的 `docker-compose.yaml`。

### TTFT 特别长，超过预期
- 确认 `--attention-backend FLASHINFER`（prefill 快）
- 128k prompt 的 TTFT 本来就 ~100s，这是 122B 模型单机 prefill 的物理上限
- 长 prompt 频繁出现时考虑开 `--enable-prefix-caching`（相同前缀复用 KV，砍 TTFT）

### 显存不足 / OOM
降 `--gpu-memory-utilization` 到 `0.80`，或降 `--max-model-len` 到 `32768`（交互场景够用）。

### 客户端流式拿不到 TTFT
qwen3 思考内容走 `delta.reasoning` 字段，不是 `delta.content`。客户端要两个字段都计数。或直接禁用 thinking（见 5.3）。

## 9. 客户端接入

### OpenAI SDK

```python
from openai import OpenAI
client = OpenAI(base_url="http://192.168.130.12:30000/v1", api_key="none")
r = client.chat.completions.create(
    model="qwen3.5-122b-int4",
    messages=[{"role":"user","content":"写一段 Python 读 CSV 的代码"}],
    extra_body={"chat_template_kwargs": {"enable_thinking": False}},
)
print(r.choices[0].message.content)
```

### Anthropic 兼容

vLLM 也暴露 `/v1/messages`，可用 Anthropic SDK 直接调用。

### Open WebUI

在已有 Open WebUI 的 OpenAI 连接中添加 `http://192.168.130.12:30000/v1` 即可。
